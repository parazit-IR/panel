After this task, the application must be able to:

Before accepting a file, validate:

* User exists.
* Payment exists.
* Payment belongs to User.
* Order belongs to User.
* Payment method is CARD_TO_CARD.
* Payment is WAITING_FOR_PAYMENT or compatible receipt-retry state.
* Payment is not APPROVED.

* Payment is not EXPIRED.
* Active manual instruction exists.
* Instruction belongs to Payment.
* Instruction is ACTIVE.
* Instruction is not expired.

* reject submission;

* safe 400 or 409 response;

Do not infer amount from receipt image.

# 23. Submission transaction boundaries

File storage and database cannot participate in one atomic transaction.

Use a compensation-based workflow.

Preferred flow:

Phase A — prepare database transaction:

* validate User, Payment, instruction;
* handle request-ID replay;
* claim one active receipt using status UPLOADING;
* commit.

Phase B — inspect and store file outside database transaction:

* sanitize filename;
* inspect content;
* calculate SHA-256;
* detect duplicate;
* store file;
* receive storage metadata.

Phase C — completion database transaction:

* reload and lock receipt;
* persist storage metadata;
* mark SUBMITTED;
* mark instruction RECEIPT_PENDING;
* mark Payment RECEIPT_SUBMITTED;
* queue receipt;
* mark Payment WAITING_FOR_REVIEW;
* commit.

On Phase B failure:

* mark receipt INVALID_FILE or FAILED-equivalent;
* no instruction/Payment transition;
* remove partial file.

On Phase C database failure after file storage:

* attempt storage compensation delete;
* if delete fails:

  * log orphan storage key safely;
  * keep operational recovery record;
  * do not expose path.

Do not hold a database transaction open during file IO.

Do not rely on self-invoked `@Transactional`.

Create focused transaction components.

# 24. Receipt queueing behavior

After successful storage:

1. Receipt becomes SUBMITTED.
2. Receipt becomes QUEUED_FOR_REVIEW.
3. Instruction becomes RECEIPT_PENDING.
4. Payment becomes RECEIPT_SUBMITTED.
5. Payment becomes WAITING_FOR_REVIEW.
6. Set reviewQueuedAt.

These transitions should be completed in one short database transaction.

Do not approve or reject.

Do not provision VPN.
