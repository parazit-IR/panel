# 20. Telegram update processor

Create:

ProcessTelegramUpdateUseCase

Implementation:

TelegramUpdateProcessor

Behavior:

1. Validate update ID.
2. Persist or claim processed-update record.
3. If already PROCESSED:

   * return idempotently;
   * do not resend messages.
4. If another worker is PROCESSING:

   * skip or return conflict internally.
5. Reject updates from bot actors.
6. Reject non-private credential-delivery flows.
7. Map update to command or callback handler.
8. Execute handler outside the claim transaction.
9. Send Telegram response outside DB transaction.
10. Mark update PROCESSED after successful intended response.
11. On retryable Telegram failure:

* mark FAILED;
* preserve retry metadata.

12. On non-retryable invalid update:

* mark PROCESSED with ignored handler.

13. On repeated failure:

* mark DEAD after configured attempts.

Do not wrap the complete handler and Telegram network call in one DB transaction.

# 22. Response uncertainty

Telegram send requests may time out after Telegram accepted the message.

This creates an uncertain result.

Choose and document a policy.

Recommended Task 35 policy:

* generate a deterministic response fingerprint from:

  * updateId;
  * handler action;
  * chatId;
  * safe payload metadata;
* store fingerprint;
* on uncertain send:

  * mark FAILED with `TELEGRAM_SEND_UNKNOWN`;
  * retry conservatively;
  * acknowledge that Telegram Bot API does not provide a general client-defined idempotency key for sendMessage.

To reduce duplicate sensitive delivery:

* prefer editing an existing menu message where possible;
* for QR and credential delivery, require explicit button action;
* on timeout after send, do not immediately retry sensitive content blindly unless policy allows a possible duplicate;
* mark the update for manual or bounded retry strategy.

Document that exactly-once outbound delivery cannot be guaranteed.

Never claim exactly-once Telegram sending.

# 33. Raw token delivery limitation

Task 33 correctly stores only a token hash.

Therefore a previously created Subscription URL cannot be reconstructed later.

Task 35 must not weaken this security rule.

Implement one of these strategies:

Option A — token rotation on delivery:

* When user asks for Subscription link or QR, explicitly rotate the token.
* Receive the new raw token once.
* Build Subscription URL.
* Send URL or QR.
* Old token becomes invalid.

Option B — encrypted secret vault:

* Store encrypted raw token.
* Not recommended because no secret-vault requirement exists yet.

Preferred Task 35 strategy:

Option A — explicit token rotation.

UX:

Button:

“Generate new subscription link”

Warning:

“Generating a new link will invalidate the previous link.”

Confirmation callback:

CONFIRM_ROTATE_SUBSCRIPTION_TOKEN

Then:

1. Call RotateSubscriptionTokenUseCase.
2. Receive raw token.
3. Build trusted Subscription URL.
4. Send URL and/or QR.
5. Never persist raw token in Telegram module.
6. Do not include token in callback data.
7. Do not log it.

Document this clearly.
