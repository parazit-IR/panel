# 38. Concurrency

Handle:

* QR generation concurrent with token rotation;
* QR generation concurrent with revocation;
* VLESS QR concurrent with provision disable;
* multiple simultaneous QR requests;
* property/config reload if project supports it.

Rules:

* Subscription URL token must be validated against current committed hash.
* After rotation commit, old token QR request must fail.
* After revocation commit, QR request must fail.
* A request that validated before a concurrent commit may finish with the previous state; minimize this window through one short read transaction and document behavior.
* Do not hold database locks while rendering image.
* Use:

  * Phase A: validate and build immutable payload;
  * Phase B: render outside transaction.
* No JVM global lock.
* QR generator must be thread-safe or stateless.

---

# 39. Transaction boundaries

Subscription URL QR:

Phase A, short transaction:

* load Subscription;
* verify ownership;
* verify token;
* validate state;
* build immutable trusted URL.

Phase B, outside transaction:

* generate QR bytes.

VLESS QR:

Phase A, short transaction or existing resolver boundary:

* validate ownership/state;
* create immutable config model.

Optional remote metadata retrieval:

* outside database transaction.

Phase B:

* render URI;
* generate QR.

Do not keep a transaction open during:

* 3x-ui call;
* QR matrix generation;
* PNG encoding;
* SVG generation;
* response streaming.
