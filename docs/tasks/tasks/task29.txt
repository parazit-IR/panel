# Task 29 — Manual Card-to-Card Payment with Unique Payable Amount

## Goal

Implement the manual card-to-card payment initiation flow.

After this task, the application must be able to:

* create a manual card-transfer payment instruction;
* generate a temporary unique payable amount;
* reserve that amount for one active Payment;
* display safe bank-card information;
* expire unused payment instructions;
* recreate or reissue an instruction safely;
* prevent two active users from receiving the same payable amount;
* preserve the original Plan and Order amount;
* remain idempotent under repeated and concurrent requests.

This unique amount is only a matching aid.

# 11. Active amount reservation design

The unique payable amount must be unique only among active, non-expired manual instructions.

Preferred PostgreSQL design:

A partial unique index on:

payable_amount

for active/reserved statuses, such as:

status IN ('CREATED', 'ACTIVE', 'RECEIPT_PENDING')

Requirements:

* Expired and cancelled instructions may reuse the amount later.
* COMPLETED instructions no longer need active reservation, but historical value remains stored.
* Do not create global uniqueness across all historical rows.
* Do not rely only on repository existence checks.
* The partial unique index is the correctness authority.

If currency may expand later, use:

(currency, payable_amount)

for uniqueness.

Preferred:

unique active reservation by currency and payable amount.

# 22. Suffix generation algorithm

Algorithm:

1. Load Payment base amount.
2. Generate suffix in configured range.
3. Compute payableAmount using Math.addExact.
4. Attempt insert.
5. If partial unique constraint conflicts:

   * generate another suffix;
   * retry up to maxGenerationAttempts.
6. If all attempts fail:

   * throw ManualPaymentAmountReservationException.
7. Never fall back to a predictable suffix.
8. Never loop forever.

Do not rely only on:

existsByPayableAmount

A pre-check may reduce conflicts but does not ensure correctness.

Catch only the relevant integrity violation.

Be careful with rollback-only transaction state.

Preferred:

Each reservation attempt should use a technically correct new transaction or a database-native insert/claim method.

Explain the chosen strategy.
