4. Selection status

Create:

PlanSelectionStatus

Initial values:

ACTIVE
CLEARED
EXPIRED
CONSUMED

Persist with EnumType.STRING.

Semantics:

ACTIVE:
The selection may be used to create an order later.

CLEARED:
The user explicitly cleared the selection.

EXPIRED:
The selection passed expiresAt.

CONSUMED:
A future order flow has consumed the selection.

Task 19 behavior:

* Create ACTIVE selections.
* Allow clear operation.
* Detect and mark expiration when selection is loaded or replaced.
* Do not implement CONSUMED transitions from an Order yet.
* A future task may consume it.

Do not physically delete selections merely because they were cleared or expired unless the selected architecture explicitly chooses a simpler current-selection table.

5. Current-selection storage design

Choose one clean design.

Preferred design:

Maintain historical selection records, but enforce only one ACTIVE selection per user using a PostgreSQL partial unique index:

UNIQUE (user_id) WHERE status = 'ACTIVE'

Benefits:

* preserves history;
* supports future audit/debugging;
* allows replace by clearing the current row and creating a new row;
* prevents multiple active selections under concurrency.

Alternative:

One mutable row per user with a unique user_id.

Choose the simpler option if the current project strongly avoids history.

Preferred final design for this task:

Historical rows with exactly one ACTIVE row per user.

Explain the decision before implementation.

SelectPlanService

Algorithm:

1. Load user.
2. Load ACTIVE plan.
3. Load current ACTIVE selection.
4. If current selection is expired:

   * mark EXPIRED;
   * persist;
   * treat as no active selection.
5. If current non-expired selection references the same plan:

   * return it;
   * newlySelected=false;
   * do not create another row.
6. If current non-expired selection references a different plan:

   * persist transition.
7. Create a new snapshot selection using current Plan data.
8. Save it.
9. Return newlySelected=true.

Requirements:

* selecting the same plan is idempotent;
* selecting a different plan replaces the current selection;
* Plan updates after selection do not alter old snapshot data;
* blocked/suspended behavior must be explicitly decided.

Recommended:

Blocked users cannot select a plan.
Suspended users cannot select a plan.
Inactive users cannot select a plan.

Create or reuse a safe exception mapped to 403 or 409.

Choose behavior consistent with current User semantics and document it.

24. Concurrency strategy

Handle concurrent selection requests for the same user.

Scenarios:

Same user selects the same plan concurrently:

* exactly one ACTIVE selection;
* both requests should ideally return the same active selection;
* no raw database exception escapes.

Same user selects different plans concurrently:

* exactly one ACTIVE selection remains;
* one selection wins deterministically according to transaction completion;
* the other request may return conflict or recover by reloading the winner.

Choose and document one behavior.

Recommended:

* database partial unique index is final authority;
* on active-selection uniqueness conflict, reload current active selection;
* if it matches requested plan, return idempotently;
* if it differs, return PlanSelectionConflictException mapped to 409.

Do not use synchronized or JVM locks.

Be careful with rollback-only transaction state after a persistence exception.

Use a correct strategy such as a separate transaction boundary or focused insert component.
