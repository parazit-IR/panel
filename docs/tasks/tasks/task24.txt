# 16. Uncertain-result handling

This is critical.

A remote create request may time out after 3x-ui has already created the client.

Implement a safe recovery strategy.

Preferred strategy:

1. Create local provision record with stable:

   * remoteClientId
   * remoteEmail
   * subscriptionId
2. Mark PROVISIONING.
3. Send create request.
4. On confirmed success:

   * mark ACTIVE.
5. On confirmed remote rejection:

   * mark FAILED.
6. On timeout or connection interruption after request may have been transmitted:

   * mark UNKNOWN.
7. Attempt to verify remote existence using inbound discovery/client lookup.
8. If client exists with matching clientId or email:

   * mark ACTIVE.
9. If confirmed absent:

   * retry creation in a new bounded attempt.
10. If still uncertain:

* return a safe provisioning-pending/unknown error.

To support verification, extend the remote read capability carefully:

Add to XuiInboundClient or a focused client:

Optional<XuiClientSnapshot> findClient(
long inboundId,
String clientId,
String email
)

Implementation may retrieve the inbound list and search its client snapshots.

Do not create a second duplicate remote client.

Do not blindly retry POST after timeout without checking existence.

# 24. Concurrency safety

Concurrent requests for the same planSelectionId:

* exactly one local provision row;
* exactly one stable remoteClientId;
* ideally exactly one remote create POST;
* other requests return or wait/reconcile;
* no raw unique-constraint error escapes.

Do not use JVM-only locks as the sole mechanism.

Use database uniqueness plus correct transaction recovery.

A local in-process lock may be an optimization, not the correctness mechanism.

Concurrent requests across multiple application instances must remain safe.

Possible strategy:

* unique planSelectionId;
* atomic status transition from PENDING to PROVISIONING;
* only the transaction that successfully claims PROVISIONING sends the POST;
* others reload and reconcile.

Consider a conditional update or pessimistic locking only if justified.

Do not over-engineer with Kafka or distributed locks.

* first POST times out;
* subsequent inbound lookup shows client exists;
* provision becomes ACTIVE;
* no second create POST.

* first result uncertain;
* lookup confirms absent;
* one bounded retry may occur;
* success marks ACTIVE.

Persistent uncertainty:

* lookup cannot confirm;
* provision remains UNKNOWN;
* safe exception returned;
* selection remains unconsumed;
* no uncontrolled retries.
