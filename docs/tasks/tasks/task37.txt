# 30. Duplicate Telegram file behavior

Use several layers:

1. Telegram processed update ID.
2. Conversation-state current attempt.
3. receiptRequestId.
4. upload-attempt unique constraint.
5. Task 30 receipt request-id idempotency.
6. active receipt unique constraint.
7. receipt SHA-256 duplicate policy.

Behavior:

* same update replay:

  * no second download/submission.
* same Telegram file sent in a new update for same Payment:

  * detect file-unique-ID hash if available;
  * if prior attempt SUBMITTED:

    * return existing success;
    * do not download again.
* same bytes with different Telegram file IDs:

  * Task 30 content hash handles it.
* same file for another User:

  * do not automatically infer fraud;
  * follow Task 30 duplicate-hash policy.

# 31. Unknown-result behavior

Possible uncertainty:

* file stored and receipt submitted;
* completion transaction fails;
* Telegram response times out.

Recovery:

1. On retry, lookup attempt by receiptRequestId.
2. Lookup receipt by receiptRequestId.
3. If receipt exists and is queued:

   * mark attempt SUBMITTED;
   * reconcile state/session;
   * send success response without re-upload.
4. If attempt says SUBMITTING but no receipt:

   * follow Task 30 compensation/orphan rules;
   * retry safely with same request ID only if semantics permit.
5. Never create another active receipt blindly.

Create:

ReconcileTelegramReceiptUploadAttemptService

if needed.
