9. Settings creation strategy

User settings must exist for every registered user.

Choose and implement one clear strategy.

Preferred strategy:

* RegisterUserService ensures default UserSettings exists after a new user is created.
* Existing users without settings receive defaults lazily when settings are requested or updated.

Requirements:

* Registration remains idempotent.
* Existing users must not receive duplicate settings.
* Database unique constraint is the final authority.
* Do not publish events unless the project already has an event pattern.
* Do not create a complex event-driven mechanism for this task.
* Explain the chosen strategy before implementation.

A valid flow:

New user registration:

1. create User;
2. save User;
3. create default UserSettings;
4. save settings;
5. return registration result.

Existing user registration:

1. load User;

2. ensure settings exist;

3. create defaults only if missing;

4. preserve existing settings.

5. Concurrency handling

Handle concurrent default-settings creation.

Requirements:

* Do not rely only on existsByUserId.
* Preserve unique user_id constraint.
* Concurrent attempts must not create duplicate rows.
* Reuse a technically correct duplicate-recovery strategy consistent with Task 10.
* Do not use synchronized or JVM-only locks.
* Do not catch generic Exception.
* Do not expose database constraint names.
