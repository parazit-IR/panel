5. Language code resolver

Reuse the existing UserLanguageResolver from Task 10.

Review and refactor it if needed.

Required behavior:

* "fa", "fa-IR", and codes starting with "fa" map to FA.
* "en", "en-US", and codes starting with "en" map to EN.
* Matching is case-insensitive.
* Trim input.
* For registration:

  * null, blank, or unsupported values default to FA.
* For explicit user language changes:

  * null, blank, or unsupported values must be rejected.

Do not create duplicate resolvers.

Provide separate methods if needed, for example:

UserLanguage resolveOrDefault(String languageCode)

UserLanguage resolveRequired(String languageCode)

Choose names consistent with the existing codebase.

8. Change language service

Create:

ChangeUserLanguageService

Suggested package:

com.parazit.panel.application.user

Requirements:

* Implement ChangeUserLanguageUseCase.
* Depend on:

UserRepository
UserLanguageResolver
SystemClockPort

* Use constructor injection.
* Define the write transaction at the application-service level.
* Validate the language code using strict resolution.
* Load the user by telegramUserId.
* Throw UserNotFoundException when absent.
* Call the domain method:

user.changeLanguage(...)

* Record the interaction time only if consistent with the current User aggregate design.
* Save through UserRepository if required.
* Return UserLanguageResult.

Behavior:

* Changing to a different supported language succeeds.
* Changing to the same language is idempotent.
* Same-language requests must not fail.
* Do not alter username, profile names, status, or blocked state.
* Do not reactivate or unblock users.
* Do not use the registration defaulting rule for explicit changes.
