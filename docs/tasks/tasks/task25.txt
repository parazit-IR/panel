# 11. Disable behavior

Disable operation rules:

ACTIVE provision:

* verify remote client;
* mark DISABLING locally;
* send remote disable request;
* confirm remote state;
* mark DISABLED.

DISABLED provision:

* return idempotent result;
* no remote update required unless optional reconciliation detects inconsistency.

DELETED provision:

* do not fail as a server error.
* Return a safe already-absent/deleted result or conflict based on API semantics.

Recommended:

Disable on DELETED returns 409 because a deleted client cannot be disabled.

UNKNOWN/FAILED provision:

* reconcile remote existence and enabled state before continuing.

Remote client already disabled:

* mark local DISABLED if needed;
* return idempotently.

Remote client absent:

Choose documented behavior.

Recommended:

* mark local UNKNOWN unless deletion was previously requested;
* return a safe conflict/not-found rather than pretending disable succeeded.

# 12. Delete behavior

Permanent deletion must require an explicit operation.

Rules:

ACTIVE:

* direct delete may be allowed.
* Mark DELETING.
* Delete remotely.
* Confirm absence.
* Mark DELETED.

DISABLED:

* preferred normal path to deletion.
* Mark DELETING.
* Delete remotely.
* Confirm absence.
* Mark DELETED.

DELETED:

* idempotent.
* Return existing DELETED state.
* No remote call required except optional reconciliation.

Remote already absent:

* mark DELETED.
* Return idempotently.

PENDING/PROVISIONING:

* deletion must fail with conflict.
* Do not delete a provisioning operation in progress.

UNKNOWN:

* reconcile before delete.
* If found, continue delete.
* If absent, mark DELETED.
* If still uncertain, keep UNKNOWN and return safe failure.

Do not physically delete the local database row.

# 19. Concurrency

Concurrent disable requests:

* exactly one operation claims DISABLING;
* at most one remote disable request where possible;
* all callers converge on DISABLED;
* no raw optimistic/persistence exception escapes.

Concurrent delete requests:

* exactly one operation claims DELETING;
* at most one remote delete request where possible;
* all callers converge on DELETED.

Disable racing with delete:

Choose deterministic behavior.

Recommended:

* DELETING has precedence.
* If delete claims first, disable returns conflict or final DELETED state.
* If disable claims first, delete may wait/reconcile and continue after DISABLED.
* Never leave an impossible state.
* Never re-enable the client.

Do not rely solely on synchronized or JVM locks.

Use database transition claims and reconciliation.

# 40. Optional manual verification

Do not run automatically.

Use only a dedicated test client on a non-production inbound.

Manual flow:

1. Provision a test client.
2. Disable it.
3. Verify in 3x-ui:

   * client still exists;
   * enable=false;
   * UUID, email, traffic, expiry, and IP limit preserved.
4. Repeat disable.
5. Confirm no corruption.
6. Delete the disabled test client.
7. Verify it no longer exists remotely.
8. Repeat delete.
9. Confirm idempotent local result.
10. Verify local audit record remains.

Do not test permanent deletion on a production user.

Do not commit real responses, URLs, cookies, client IDs, or keys.
