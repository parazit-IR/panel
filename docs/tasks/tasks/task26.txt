# 5. Renewal semantics

Define renewal explicitly.

Create two renewal modes:

EXTEND_FROM_CURRENT_EXPIRY

EXTEND_FROM_NOW

Create:

RenewalMode

Rules:

EXTEND_FROM_CURRENT_EXPIRY:

* If the current expiry is still in the future:
  newExpiry = currentExpiry + requested duration
* If already expired:
  newExpiry = now + requested duration

EXTEND_FROM_NOW:

* newExpiry = now + requested duration regardless of current expiry

Default recommended mode:

EXTEND_FROM_CURRENT_EXPIRY

Requirements:

* Duration must be positive.
* Use calendar-independent day duration consistent with Plan duration rules.
* Use Instant and Duration or ChronoUnit.DAYS carefully.
* No local timezone calculations.
* Do not shorten expiry through a renewal command.
* Repeating the same renewal request must not accidentally extend twice.

Because renewal is not naturally idempotent, require an explicit operation ID.

# 13. Complete update payload strategy

Some versions of 3x-ui require a full client object during update.

Implement read-modify-write:

1. Load remote client by inbound ID and stable client UUID.
2. Verify remote email against local record.
3. Copy all existing required remote values.
4. Change only explicitly requested fields.
5. Serialize the complete update payload.
6. Submit update.
7. Reload remote state.
8. Verify intended changes.

Never build an update from only local fields if that could clear remote-only fields.

Never send null/default values for fields that should remain unchanged.

Never change remote client UUID.

# 29. Transaction boundaries

Do not keep a database transaction open during remote calls.

Use three-stage orchestration:

Phase A — prepare transaction:

* validate user/provision;
* resolve/create operation;
* detect idempotent replay;
* claim operation IN_PROGRESS;
* commit.

Phase B — remote call:

* load remote state;
* execute update/reset;
* verify remote state.

Phase C — completion transaction:

* update local provision;
* mark operation SUCCEEDED/FAILED/UNKNOWN;
* commit.

Create focused components such as:

PrepareXuiClientOperationTransaction

CompleteXuiClientOperationTransaction

LoadXuiClientOperationTransaction

Do not use self-invoked @Transactional methods.

Do not duplicate one transaction class per use case unless necessary.

A reusable, focused operation coordinator is acceptable.

# 31. Idempotency behavior

When operationId already exists:

Same fingerprint:

* If SUCCEEDED:
  return existing final result.
* If IN_PROGRESS:
  reconcile or return documented in-progress behavior.
* If UNKNOWN:
  reconcile remote state.
* If FAILED:
  return existing failure or allow explicit new operation ID.
* Do not execute the operation a second time blindly.

Different fingerprint:

* throw XuiOperationIdConflictException.
* Map to HTTP 409.

Particularly verify ADD_TRAFFIC cannot apply twice.

# 32. Concurrency

Concurrent identical operation IDs:

* one operation row;
* one remote update/reset where possible;
* callers converge on same result;
* no raw unique error.

Concurrent different operation IDs on same provision:

Choose a safe serialization policy.

Recommended:

Allow only one IN_PROGRESS client operation per provision.

Create a partial unique index conceptually:

UNIQUE (provision_id) WHERE status = 'IN_PROGRESS'

If PostgreSQL and current schema strategy support it.

Alternative:

Use pessimistic row locking during claim.

Do not lock during remote HTTP call.

Recommended implementation:

* operation claim establishes one active operation;
* competing operation receives 409 ClientOperationInProgress;
* caller may retry later.

Do not allow simultaneous renewal and deletion.

Task 25 delete/disable operations should integrate into the same operation coordination if practical.

Avoid a total architecture rewrite.

# 34. Uncertain-result reconciliation

Update timeout:

* reload remote client;
* compare requested fields;
* if desired state is present:
  mark SUCCEEDED;
* if old state remains:
  one bounded retry may occur;
* if remote lookup unavailable:
  mark UNKNOWN;
* do not blindly submit repeatedly.

Reset timeout:

* reload traffic;
* if zero:
  mark SUCCEEDED;
* if not zero:
  one bounded retry;
* if unavailable:
  UNKNOWN.

Add-traffic timeout:

* compare remote configured limit to expected exact new value.
* If expected value present:
  succeeded.
* If old value present:
  bounded retry.
* If any unexpected third value appears:
  mark conflict/UNKNOWN.
* Never recalculate from the new remote value and add again automatically.

Renew timeout:

* compare remote expiry to exact calculated expected expiry.
* Do not extend again if expected expiry is already present.
