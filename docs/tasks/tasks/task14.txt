4. Referral code ownership

Every user must have a stable referral code.

Choose one clean design.

Preferred design:

Add a referralCode field to User.

Requirements:

* String referralCode
* required
* unique
* immutable after assignment
* normalized uppercase
* no spaces
* short enough for Telegram deep links
* not derived directly from sequential identifiers
* generated once for each user
* existing users without a code must receive one safely

Suggested code format:

8 to 12 uppercase alphanumeric characters

Allowed characters should avoid ambiguous characters where practical.

ABCDEFGHJKLMNPQRSTUVWXYZ23456789

O
1
I

Referral code requirements:

* Must be case-insensitive on input.
* Must be stored normalized.
* Must be unique at database level.
* Must not expose the user's Telegram ID.
* Must not be manually editable.
* Must not be regenerated during profile updates or repeated registration.
* Must remain stable once created.

Create a dedicated ReferralCode entity.

9. Existing-user backfill strategy

Preferred:

Only choose this if uniqueness, security, and predictability are properly handled.

Do not create insecure sequential referral codes.

41. Verification commands

Run the commands appropriate to the current build tool.

For Gradle:

./gradlew clean test
./gradlew build

For Maven:

./mvnw clean test
./mvnw verify

Also run the application with the local profile and PostgreSQL.

Manual verification:

1. Register two users.
2. Get the first user's referral overview.
3. Use the first user's referral code for the second user.
4. Repeat the same assignment.
5. Try assigning a different referral code to the second user.
6. Try self-referral.

Confirm:

* all previous tests pass;
* Flyway succeeds;
* Hibernate validation succeeds;
* each user has one stable referral code;
* each referred user has at most one referrer;
* repeated same assignment is idempotent;
* conflicting assignment returns 409;
* no rewards are created;
* no Telegram integration was added;
* no credentials were committed.
