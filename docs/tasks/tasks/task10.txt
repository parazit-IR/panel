6. Registration service

Create:

RegisterUserService

Suggested package:

com.parazit.panel.application.user

Requirements:

* Implement RegisterUserUseCase.
* Depend on:

UserRepository
SystemClockPort
UserLanguageResolver

* Use constructor injection.
* Define the transaction boundary at the application service.
* Use @Transactional on the registration operation.
* Do not annotate the interface with @Transactional.
* Use the current time from SystemClockPort.
* Do not call Instant.now() directly.
* Do not inject infrastructure implementations directly.

Registration algorithm:

1. Validate the command is not null.
2. Validate telegramUserId is positive.
3. Validate firstName is not blank.
4. Resolve UserLanguage from languageCode.
5. Search by telegramUserId.
6. If the user does not exist:

   * create User through its controlled factory;
   * save it;
   * return result with newlyCreated=true.
7. If the user already exists:

   * update Telegram profile information;
   * record the latest interaction time;
   * preserve the existing language selection;
   * do not overwrite language during repeated registration;
   * save only if required by the repository/JPA design;
   * return result with newlyCreated=false.

Important behavior:

* Registration must be idempotent.
* Repeating the same command must not create duplicate users.
* Re-registering an existing user must update username, first name, last name, and lastInteractionAt.
* Re-registering must not automatically unblock a blocked user.
* Re-registering must not activate, unsuspend, or change the user's business status.
* Re-registering must not overwrite the user-selected language.
* New users receive the resolved initial language.
* The database unique constraint remains the final protection against duplicates.

7. Concurrency and duplicate protection

Handle concurrent registration attempts for the same telegramUserId.

Requirements:

* Do not rely only on existsByTelegramUserId.
* Use findByTelegramUserId as the primary read.
* Keep the database unique constraint.
* Handle a race where two transactions both attempt to create the same user.
* Catch only the relevant Spring data integrity exception at the application/infrastructure boundary where appropriate.
* After a duplicate-key conflict, load the existing user and return it as newlyCreated=false.
* Do not expose raw SQL or constraint names.
* Do not catch generic Exception.
* Avoid retry loops without limits.
* Document the concurrency strategy.

A valid strategy is:

1. check existing user;
2. attempt save;
3. on unique constraint violation, load the existing user;
4. update the Telegram profile if needed;
5. return existing user.

Be careful that transaction state may be marked rollback-only after a persistence failure.

Choose a technically correct Spring transaction strategy.

Acceptable approaches include:

* a separate REQUIRES_NEW creation component;
* database upsert implemented in a focused repository method;
* retrying the complete registration operation through a dedicated transaction boundary;
* another clean solution compatible with the existing architecture.

Explain the selected approach before implementation.

Do not use synchronized or JVM-only locks because the application may run on multiple instances.
