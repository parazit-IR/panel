# 19. Initiation transaction boundaries

Do not keep a DB transaction open during gateway request.

Preferred flow:

Phase A — prepare transaction:

* validate Payment/User/Order;
* handle requestId replay;
* create ZarinpalPaymentAttempt in CREATED;
* set gatewayAmount;
* mark REQUESTING;
* commit.

Phase B — remote request:

* call Zarinpal;
* receive authority or failure.

Phase C — completion transaction:

* on success:

  * mark attempt REDIRECT_READY;
  * persist authority;
  * mark Payment WAITING_FOR_PAYMENT;
* on confirmed gateway rejection:

  * mark attempt FAILED;
  * keep Payment retryable according to domain rules;
* on uncertain timeout:

  * mark attempt UNKNOWN;
  * do not mark Payment APPROVED.

Use focused transaction components.

Do not rely on self-invoked `@Transactional`.

# 24. Callback cancellation behavior

* If Payment already APPROVED:

* If attempt already VERIFIED:

* Otherwise:

  * mark Payment CANCELLED if domain rules permit;
  * store no raw gateway message.
* Repeated cancelled callback is idempotent.
* No Verify call.
* No VPN provisioning.

Unknown callback status:

* Do not treat as approved.
* Return failed/unknown safely.
* Log sanitized warning.

# 28. Uncertain verification handling

Scenario:

Verify request times out after Zarinpal may have processed it.

Rules:

* Do not approve.
* Mark attempt UNKNOWN.
* Preserve authority and amount.
* Do not create another payment attempt immediately.
* Allow a bounded reconciliation Verify call using the same authority and amount.
* If later Verify returns success/already-verified:
  approve idempotently.
* If repeated verification remains unavailable:
  leave UNKNOWN.
* Expose a safe retryable result.
* Future scheduled reconciliation may be added later.

Do not create a new authority for an attempt whose payment outcome is unknown.

# 29. Initialization uncertainty

A payment request timeout may occur after Zarinpal generated an authority but before the application received it.

Because the gateway request API may not support a merchant-provided idempotency key:

* do not blindly retry indefinitely;
* mark the attempt UNKNOWN;
* do not create unlimited new attempts;
* allow the user/operator to retry through a new requestId only after documented policy;
* expire uncertain attempts after a configured interval if later needed;
* never mark Payment approved.

Document this limitation.
