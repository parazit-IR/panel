2. Repository abstraction

Create a generic repository abstraction outside the infrastructure layer.

Suggested package:

com.parazit.panel.domain.repository

Create:

BaseRepository<T, ID>

Requirements:

* Use generics.
* Keep it independent from Spring Data.
* Do not extend JpaRepository, CrudRepository, PagingAndSortingRepository, or any Spring interface.
* Define only useful methods.

Include:

Optional<T> findById(ID id)

List<T> findAll()

T save(T entity)

List<T> saveAll(Collection<T> entities)

boolean existsById(ID id)

long count()

void delete(T entity)

void deleteById(ID id)

Do not add pagination, sorting, specifications, or query-by-example yet.

Do not add methods that are not currently justified.

5. Generic repository adapter

Create a reusable adapter implementation.

Suggested package:

com.parazit.panel.infrastructure.persistence.repository

Create:

JpaRepositoryAdapter<T, ID>

Requirements:

* Implement BaseRepository<T, ID>.
* Delegate operations to SpringDataRepository<T, ID>.
* Use constructor injection.
* Do not use field injection.
* Do not annotate the generic adapter with @Repository if doing so causes invalid generic bean registration.
* Keep exception translation handled by Spring Data.
* Do not catch generic exceptions unnecessarily.
* Do not leak Spring Data types into domain interfaces.

Implement all BaseRepository methods.

For saveAll:

* Accept Collection<T>.
* Return List<T>.
* Preserve expected behavior.
* Do not silently ignore nulls.

8. Transaction behavior

Document and verify transaction responsibilities.

Rules:

* Repository adapters should not define broad business transactions.
* Transaction boundaries belong primarily in application services.
* Repository methods may rely on Spring Data transactional behavior.
* Do not add @Transactional to every method without justification.
* Read-only transaction optimization may be discussed but should not be overused in this task.
