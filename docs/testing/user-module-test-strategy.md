# User Module Test Strategy

## Scope

This strategy covers the Phase 2 user module: user registration, profile,
language, settings, referral code generation, and referral assignment. It does
not cover plans, payments, subscriptions, Telegram bot handlers, 3x-ui, VPN
provisioning, admin dashboard, or reminder jobs.

## Test Pyramid

Domain tests are pure JUnit tests with no Spring context. Application service
tests use deterministic fakes and fixed instants. Repository, use-case,
controller, and concurrency tests use real PostgreSQL through Testcontainers.

## Domain Unit Tests

Domain tests cover `User`, `UserSettings`, and `Referral` creation, validation,
normalization, state transitions, idempotency, and immutability rules. They do
not use Spring, JPA repositories, or PostgreSQL.

## Application Unit Tests

Application tests cover command/query validation, missing resources,
idempotency, field preservation, clock usage, referral-code backfill, referral
overview, and referral assignment behavior. Fakes are deterministic and avoid
over-specifying Spring or JPA implementation details.

## Repository Integration Tests

Repository tests use PostgreSQL and Flyway-managed schema validation. They
verify repository adapter behavior, UUID persistence, auditing, business lookup
methods, enum persistence, foreign keys, unique constraints, and check
constraints.

## Use-Case Integration Tests

Use-case integration tests start the Spring context and call application ports.
They verify registration, profile, language, settings, and referral workflows
with real transaction boundaries and persistence.

## Controller Tests

Controller tests use MockMvc against the internal endpoints and assert status
codes, JSON content type, field names, ISO-8601 timestamps, validation errors,
standard error responses, trace IDs, malformed JSON handling, and absence of
JPA entity internals.

## Concurrency Tests

Concurrency tests use PostgreSQL, separate transactions, `ExecutorService`,
`CyclicBarrier`, and bounded timeouts. They cover concurrent registration,
settings default creation, same-referrer referral assignment, and
different-referrer referral assignment. They do not use `Thread.sleep`.

## Testcontainers Strategy

`PostgreSqlContainerSupport` starts one PostgreSQL 17 container for the test JVM
and registers datasource properties through `@DynamicPropertySource`. Tests do
not require locally installed PostgreSQL or Docker Compose. Container startup
failures are not hidden.

## Database Cleanup

Full integration and controller tests call `DatabaseCleaner.cleanUserModuleTables`
before each test. It truncates `referrals`, `user_settings`, and `users` without
touching Flyway history. Repository slice tests rely on transactional rollback
unless they need targeted row removal inside a scenario.

## Fixed-Clock Strategy

Controller tests use `FixedClockTestConfiguration` where stable timestamps are
part of the API contract. Persistence audit tests use `MutableClockTestConfiguration`
to advance time deterministically. Tests must not depend on wall-clock sleeps.

## Coverage

JaCoCo reports are generated by:

```bash
env -u JAVA_HOME ./gradlew jacocoTestReport
```

Reports are written to `build/reports/jacoco/test/html/index.html` and
`build/reports/jacoco/test/jacocoTestReport.xml`. Domain and application code
should remain the primary coverage target; configuration-only classes and the
application bootstrap are lower value.

## Architecture Checks

`ArchitectureRulesTest` verifies key rules without adding ArchUnit:

- domain does not depend on API or infrastructure;
- application does not depend on infrastructure or Spring Data repositories;
- controllers do not depend on Spring Data repositories or JPA internals;
- no field injection or service-locator usage;
- Spring Data repositories stay in infrastructure;
- domain repositories do not extend Spring Data types.

## Commands

Use Java 21. In this environment, `JAVA_HOME` points to Java 11, so the commands
unset it to use the Java 21 runtime on `PATH`.

```bash
env -u JAVA_HOME ./gradlew clean test
env -u JAVA_HOME ./gradlew jacocoTestReport
env -u JAVA_HOME ./gradlew check
env -u JAVA_HOME ./gradlew build
env -u JAVA_HOME SPRING_PROFILES_ACTIVE=local ./gradlew bootRun
```

## Debugging Container Tests

If a container test fails before Spring starts, check Docker availability first.
If a test fails with a connection refused error, verify the singleton container
support was used and no class-level container stopped while a Spring context was
cached. For data contamination, check that the test calls
`DatabaseCleaner.cleanUserModuleTables` before each scenario.

## Known Limitations

Trace IDs are generated per request and are not reused from incoming
`X-Trace-Id` headers. The API still uses Spring Security defaults in tests, so
MockMvc supplies basic auth and CSRF tokens for state-changing requests.
